У проекта есть этапы, и правила одного этапа не переносятся автоматически на
следующий. Соглашения по коду в CLAUDE.md (использовать Angular, организовывать по
фичам, использовать сигналы, настраивать Storybook) — это правила **этапа реализации**. Они
ничего не говорят про этап дизайна, потому что на этапе дизайна кода ещё нет. Предложить
воркспейс Angular, пока пользователь работает в Penpot, — это не дотошность. Это
ошибка категории: она переводит разговор на этап, о котором пользователь не просил, тратит
его время и показывает, что описание задачи никто не прочитал.

В проекте на этапе дизайна (2026-04-26) случилось ровно это. Команда работала над
задачей по макету в Penpot и получила в ответ предложение по воркспейсу Angular плюс Storybook.
Реакция была прямой. Правила по Angular действительно существуют, и они не ошибочны, но это
**правила про код**, а задача по дизайну — это не задача по коду.

## Почему это важно

### Цена путаницы этапов

Когда пользователь говорит «помоги с этим макетом в Penpot», объём задачи ограничен: визуальный
дизайн, решения по компоновке, варианты компонентов, цветовые токены, типографика, потоки прототипа.
Предложение фреймворка загоняет пользователя в один из двух плохих исходов. Он может потратить время
на отказ и возвращение разговора в нужное русло — или может согласиться и оказаться не на том
этапе для той работы, которая ему действительно нужна. И то, и другое — провал. Первое сжигает его
время; второе тянет преждевременные решения по реализации на этап, где реальные
ограничения дизайна ещё неизвестны.

Решения по дизайну и решения по реализации зависят друг от друга, но принимаются они
по отдельности. Этап дизайна определяет, что строить: компоновку, визуальную иерархию, модель
взаимодействия, границы компонентов как визуальные понятия. Этап кода определяет, как это
реализовать. Смешайте эти два — и вы заставляете принимать решения по реализации до того,
как дизайн устоялся, а это путь к компоненту, который выглядит неправильно, но который уже не
изменить, потому что он намертво вшит в граф управления состоянием.

### Слова-триггеры

Задача, содержащая что-либо из перечисленного ниже, — это задача по дизайну, а не по коду, пока
явно не сказано иначе:

- макет / вайрфрейм / прототип
- дизайн (когда используется как существительное для артефакта, а не для системы)
- Penpot / Figma (включая транслитерации, которыми команда называет локальный self-hosted экземпляр Penpot)
- токены (дизайн-токены, а не токены авторизации)
- компонент в Penpot

Если эти слова появляются, ответ остаётся в рамках понятий дизайна, возможностей
дизайн-инструмента и визуальных решений. Код входит в разговор только тогда, когда пользователь
явно об этом просит.

## Как применять

Когда приходит задача по дизайну, оставайтесь в пределах области дизайн-инструмента:

**Типографика**
Обсуждайте выбор гарнитуры, шкалы насыщенности и размера, межстрочный интервал и трекинг, оптический
размер, оси вариативного шрифта. Результат: рекомендуемая таблица шкалы шрифта, JSON дизайн-токенов
W3C для этой шкалы или файл токенов, импортируемый в Penpot.

**Цвет**
Обсуждайте построение палитры (основной, семантический, нейтральный, ошибки), коэффициенты
контраста, структуру токенов для тёмной/светлой темы, ограничения бренда. Результат: файл
JSON дизайн-токенов W3C с палитрой или плоский список CSS-переменных, готовый к вставке.

**Компоновка и плотность**
Обсуждайте сеточные системы (8-точечная, 4-точечная), шкалы отступов, стратегию точек перелома, ширины
контейнеров, плотность информации. Результат: таблица токенов отступов, спецификация сетки,
аннотированные наброски компоновки в виде текста.

**Варианты компонентов**
Обсуждайте структуру компонента в Penpot (главный компонент + варианты), оси пропсов (размер, состояние,
акцент), поведение auto-layout, паттерны вложенных компонентов. Результат: спецификация сетки
вариантов — какие оси, какие значения по каждой оси, как они комбинируются — и SVG-ассеты, если
запрошены.

**Прототип и взаимодействие**
Обсуждайте связи потока в Penpot, типы переходов (мгновенный, растворение, сдвиг), значения
задержки, поведение прокрутки, фиксированные/липкие оверлеи, горячие зоны взаимодействия. Результат:
описание разводки прототипа или прямое управление Penpot через браузерный MCP.

```jsonc
// Example: W3C design tokens for a type scale — the correct deliverable
// for a typography design task; NOT a TypeScript type, NOT a Storybook story.
{
  "typography": {
    "scale": {
      "xs":   { "$value": "0.75rem",  "$type": "dimension" },
      "sm":   { "$value": "0.875rem", "$type": "dimension" },
      "base": { "$value": "1rem",     "$type": "dimension" },
      "lg":   { "$value": "1.125rem", "$type": "dimension" },
      "xl":   { "$value": "1.25rem",  "$type": "dimension" },
      "2xl":  { "$value": "1.5rem",   "$type": "dimension" },
      "3xl":  { "$value": "1.875rem", "$type": "dimension" },
      "4xl":  { "$value": "2.25rem",  "$type": "dimension" }
    },
    "weight": {
      "regular": { "$value": 400, "$type": "fontWeight" },
      "medium":  { "$value": 500, "$type": "fontWeight" },
      "semibold":{ "$value": 600, "$type": "fontWeight" },
      "bold":    { "$value": 700, "$type": "fontWeight" }
    }
  }
}
```

Обратите внимание на то, чего здесь нет: ни модуля Angular, ни истории Storybook, ни определения
интерфейса, ни декоратора компонента. Всё это относится к этапу реализации. На этапе дизайна
файл токенов и спецификация компоновки сами по себе — полные, корректные результаты.

### Переход к коду — только по запросу

Этап дизайна заканчивается, когда пользователь явно даёт сигнал: «Окей, теперь давай это соберём»,
«сгенерируй компонент», «начни проект на Angular». В этот момент, и только тогда, начинают
действовать соглашения по коду. До этого держите каждый ответ внутри пространства дизайн-инструментов.

Если артефакту дизайна (файлу токенов, спецификации компонента) понадобится реализационный
двойник, можно **отметить** это одним предложением, например «когда будешь готов к
реализации, эти токены напрямую ложатся на CSS-переменные». Не разворачивайте это в
предложение по фреймворку или структуру файлов, пока кто-то об этом не попросит.

## Антипаттерны

Все приведённые ниже ответы на задачу по макету в Penpot ошибочны, независимо от технической
корректности:

```
// Anti-pattern 1: Proposing a framework workspace
// Trigger: "help me design this mockup in Penpot"
// Wrong response: "Let's set up an Angular workspace with Storybook so we can develop
//                  the components in isolation..."
// Why wrong: the user is in Penpot; no code exists; a workspace is phase-2 work.

// Anti-pattern 2: Delivering a TypeScript interface instead of a token file
// Trigger: "define the colour tokens for the brand"
// Wrong response: export interface BrandTokens { primary: string; secondary: string; }
// Right response: a W3C design token JSON file with the palette values.

// Anti-pattern 3: Recommending a component library at the design stage
// Trigger: "how should I structure the card component variants in Penpot?"
// Wrong response: "Angular Material has a card component; you can use that as the basis."
// Why wrong: Angular Material is an implementation; the user is designing, not building.

// Anti-pattern 4: Generating Storybook stories for a design spec
// Trigger: "spec out the button variants"
// Wrong response: a .stories.ts file
// Right response: a table of variant axes (size × emphasis × state) with visual notes.
```

## Контроль

Это обеспечивает понимание прочитанного, а не линтер. Проверка простая: содержит ли описание
задачи хоть одно из приведённых выше слов-триггеров этапа дизайна? Если да, ограничьте
ответ пространством дизайн-инструментов, пока запрос на код не придёт явно. На ревью пул-реквеста
ответ на задачу по дизайну, содержащий импорт фреймворка или каркас
воркспейса, был ошибкой — независимо от того, насколько хорош код.
