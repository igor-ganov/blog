Эмуляция устройства в Chrome DevTools — это симулятор вьюпорта. Она задаёт `window.innerWidth`,
масштабирует вьюпорт и рассылает события касаний, но не воспроизводит модель скроллбаров
операционной системы, поведение методов ввода, стек отрисовки шрифтов и геометрию хрома
браузера. Поэтому вёрстка может выглядеть правильно в эмуляции DevTools и при этом нести
видимый дефект, который проявляется, только когда человек держит реальный телефон в руках.

На одном проекте (2026-04-19) это нас укусило. Объявление `scrollbar-gutter: stable` на
элементе `html`, добавленное, чтобы убрать сдвиг вёрстки при открытии модалки на десктопе,
дало видимую белую полосу у правого края каждой страницы на мобильном. Десктопные браузеры
с overlay-скроллбарами (macOS, iOS, Android) не резервируют место под жёлоб, но
`scrollbar-gutter: stable` всё равно заставляет его резервировать — и вы получаете постоянную
белую полосу шириной 15px вдоль правой стороны экрана. Мобильная эмуляция DevTools её никогда
не показывала, потому что DevTools не воспроизводит модель overlay-скроллбаров точно. Полоса
была видна только на реальном железе.

С того момента работу над мобильной вёрсткой принимали только как полную папку скриншотов
со всеми страницами в размерах мобильного вьюпорта. Одной показательной страницы мало.

## Почему это важно

### Разрыв между эмуляцией DevTools и устройством

Мобильная эмуляция DevTools корректно воспроизводит:
- Ширину и высоту вьюпорта для выбранного пресета устройства
- `device-pixel-ratio` для CSS, зависящего от плотности пикселей
- Рассылку событий касаний
- UA-строку (через панель сетевых условий)

Мобильная эмуляция DevTools НЕ воспроизводит точно:
- **Модель скроллбаров** — десктопный Chrome рисует эмуляцию с той же моделью скроллбаров,
  что и десктопная ОС. Overlay-скроллбары (iOS, Android, macOS с настройкой
  «показывать скроллбары: при прокрутке») не резервируют место в вёрстке, а `scrollbar-gutter`
  считает, что резервируют.
- **Геометрию хрома браузера** — адресная строка на мобильном сжимается при прокрутке,
  меняя `100vh` на значение больше ожидаемого. У `dvh` (динамическая высота вьюпорта) этой
  проблемы нет; у `vh` есть. DevTools не воспроизводит сворачивание адресной строки.
- **Отрисовку системных шрифтов** — iOS рисует шрифты `-apple-system` с субпиксельным
  округлением, которое отличается от Chrome на Android. Текст, который влезает в контейнер
  при 375px в DevTools, на реальном устройстве iOS может перенестись иначе.
- **Оверлей метода ввода** — программная клавиатура на мобильном уменьшает видимый вьюпорт.
  Элементы с `100vh`, лежащие над сгибом, могут перекрываться клавиатурой так, как режим
  DevTools «показать клавиатуру» не воспроизводит.

### Инцидент со `scrollbar-gutter`

`scrollbar-gutter: stable` убирает сдвиг вёрстки, когда на странице появляется контент
достаточной высоты, чтобы включить скроллбар. Жёлоб резервируется заранее, поэтому появление
скроллбара не сдвигает контент влево. На десктопе это именно то, что нужно.

На мобильном скроллбары — это оверлеи. Они появляются временно поверх контента и не резервируют
места, поэтому `scrollbar-gutter: stable` на `html` в мобильном браузере с overlay-скроллбарами
резервирует жёлоб, который ничем не заполнен. Вы видите его как белую полосу шириной с жёлоб
у правого края каждой страницы.

Лечение — ограничить свойство тем брейкпоинтом, где есть десктопные скроллбары.

```css
/* Bad: applies scrollbar-gutter reservation to all viewports including mobile */
html {
  scrollbar-gutter: stable;
}

/* Good: reserve gutter only where scrollbars take up layout space */
@media (min-width: 768px) {
  html {
    scrollbar-gutter: stable;
  }
}
```

При отладке мобильного переполнения или неожиданного пустого места справа проверяйте это
свойство в первую очередь. Это самый частый источник проблем у правого края, которые
прячутся в эмуляции и вылезают на устройстве.

### Проблема `100vh` и адресной строки

У мобильных браузеров динамический вьюпорт: адресная строка прячется при прокрутке, из-за
чего видимый вьюпорт становится выше `100vh`, как только пользователь начинает скроллить.
Элемент с `height: 100vh` оказывается ниже доступного места, когда адресная строка ушла,
оставляя зазоры под героем, футеры, которые не достают до низа, и обрезанные полноэкранные
оверлеи.

```css
/* Bad: 100vh is "initial viewport height" — shorter than full-screen on mobile after
   address bar hides */
.hero {
  height: 100vh;
}

/* Good: dvh tracks the dynamic viewport height — correct on mobile and desktop */
.hero {
  height: 100dvh;
}

/* Fallback for browsers that do not support dvh (Safari < 15.4) */
.hero {
  height: 100vh;       /* fallback */
  height: 100dvh;      /* progressive enhancement */
}
```

## Как применять

### Требование доказательства

Работа над мобильной вёрсткой завершена только тогда, когда есть папка со скриншотами,
содержащая каждую изменённую страницу, снятую на реальном мобильном вьюпорте. Структура
папки такая:

```
screenshots/mobile-proof/
  home.png
  listing-detail.png
  search-results.png
  user-profile.png
  settings.png
  ... (one screenshot per route that exists in the app)
```

«Каждая страница» означает каждый отдельный маршрут, а не только те, что PR затронул напрямую.
Изменение вёрстки (изменение сетки, отступов, шапки, CSS-переменной) может прокатиться по
страницам, которые автор сознательно не трогал, и скриншот каждой страницы — единственный
надёжный способ поймать такие каскадные регрессии.

Если снять скриншоты на реальном железе не получается, используйте мобильный браузер напрямую:
откройте страницу на физическом телефоне и снимите там либо воспользуйтесь облачным сервисом
с реальными устройствами. Не подменяйте это панелью устройств в DevTools.

### Чеклист отладки мобильной вёрстки

Когда сообщили о проблеме мобильной вёрстки или вы её подозреваете, проверяйте в таком порядке:

1. `scrollbar-gutter` на `html` или `body` — он не ограничен брейкпоинтом?
2. `height: 100vh` на полноэкранных элементах — не должно ли быть `100dvh`?
3. Элементы с `position: fixed` — учитывают ли они динамический вьюпорт?
4. `overflow-x: hidden` на `body` — он прячет реальное горизонтальное переполнение, вместо
   того чтобы его починить? (Спрятанное переполнение маскирует баги вёрстки, а не устраняет их.)
5. Метатег вьюпорта — присутствует ли `width=device-width, initial-scale=1`? Без него
   мобильные браузеры используют масштабированный layout-вьюпорт в 980px.

```html
<!-- Required — without this, the entire responsive layout breaks on mobile -->
<meta name="viewport" content="width=device-width, initial-scale=1">
```

### CSS-свойства с расхождениями эмуляция/устройство

| Свойство | Поведение в эмуляции | Поведение на реальном устройстве |
|---|---|---|
| `scrollbar-gutter: stable` | Полосы нет (в эмуляции нет модели overlay) | Белая полоса на устройствах с overlay-скроллбаром |
| `height: 100vh` | Совпадает с эмулируемым вьюпортом ровно | Ниже видимого вьюпорта, когда адресная строка прячется |
| `position: fixed` + `bottom: 0` | Стоит у низа эмулируемого вьюпорта | Стоит над клавиатурой, когда она открыта |
| `touch-action: manipulation` | В эмуляции эффекта не видно | Убирает задержку тапа в 300ms в части браузеров Android |
| `-webkit-overflow-scrolling: touch` | В эмуляции игнорируется | Включает инерционную прокрутку на iOS (устарело, но ещё работает) |

## Антипаттерны

**«Я проверил в DevTools, выглядит нормально»**

DevTools — это доказательство для десктопа в узкой ширине, а не для мобильного. Специфичные
для мобильного свойства и поведение требуют реального железа или облака с реальными устройствами,
так что скриншот из DevTools мобильным доказательством не считается.

**Починить одну страницу, не проверив все**

CSS-изменение, затрагивающее примитив вёрстки (`.container`, `.page`, `body`, `html`,
`:root`), влияет на каждую страницу. Проверка только той страницы, над которой активно
работают, пропускает каскадные регрессии. Папка скриншотов покрывает все маршруты.

**Чинить мобильное переполнение через `overflow-x: hidden` на body**

Это прячет переполнение, а не устраняет его. Дефект вёрстки остаётся на месте, просто
больше не виден. В части мобильных браузеров скрытие переполнения на `body` ещё и убивает
инерцию прокрутки и ломает элементы с `position: fixed`. Правильное лечение — найти элемент,
вызывающий переполнение, и поправить его ширину или transform.

```css
/* Anti-pattern: masks the problem */
body {
  overflow-x: hidden;
}

/* Diagnosis tool: use this temporarily to find the offending element, then remove */
* {
  outline: 1px solid red;
}
/* Then fix the actual element — do not ship the hidden overflow */
```
