Цикл разработки — это последовательность, которая переводит элемент бэклога из состояния
«готов к старту» в «смерженный PR». Он лежит на слой ниже spec-driven-процесса. Спецификация
выдаёт задачи через требования, дизайн и список задач; цикл разработки прогоняет каждую задачу
одинаково. Одинаковый прогон каждый раз убивает накладные расходы на «а что мне делать дальше?»,
и ревьюеры заранее знают, что будет внутри PR, рождённого этим циклом.

## Зачем это нужно

Единый цикл даёт единый результат. У PR, вышедшего из этого цикла, есть именованная ветка,
коммиты, которые можно проследить до задачи, тест, который начался красным и закончился зелёным,
поведение, проверенное в реальном рантайме, и документация, совпадающая с кодом. Если чего-то
из этого нет, PR читается как незаконченный, потому что чек-лист все уже знают наизусть.

Цикл к тому же закрепляет приоритеты. Документация пишется до открытия PR, а не после спринта.
Проверка сначала на десктопе — это отдельный поименованный шаг, а не то, что делаешь, если
останется время. А падающий тест в начале — это доказательство, что тест вообще способен упасть,
а только это и делает его пригодным для отлова регрессии позже.

## Как применять

### Шаг 1: соберите контекст

Прежде чем писать хоть строчку кода, достаньте:

- Элемент бэклога (задача, issue, тикет) и его полное описание.
- Родительский эпик или фазу спецификации, к которой задача относится.
- Связанные элементы: зависимые задачи, привязанные разделы дизайна, упомянутые требования.

Вы читаете не ради чтения. Вы вытаскиваете контракты, которые реализация обязана соблюсти, а
тесты — проверить: форму API, схему данных, типы событий.

### Шаг 2: ветка

```bash
git checkout -b feat/<work-item-id>-<short-title>
# Examples:
# feat/123-outbox-relay
# feat/BLOG-47-mobile-nav
```

Ветвитесь от свежего `main` (или от точки слияния родительской ветки эпика, если работаете внутри
закрытого эпика). Не переиспользуйте ветку от предыдущей задачи.

### Шаг 3: план выполнения и план тестов

Запишите (или подтвердите) план выполнения в issue задачи или в черновой заметке:

```markdown
## Execution plan
1. Add `failure_count` column migration (TASK-7, REQ-4)
2. Add promotion logic in OutboxRelay.attemptDelivery
3. Add metric emission
4. Unit test: promotion at exactly 10 failures
5. Unit test: metric emitted on promotion
6. Integration test: relay promotes real row in test DB

## Test plan
- Unit: OutboxRelay.attemptDelivery – promotes at threshold
- Unit: OutboxRelay.attemptDelivery – emits dlq.moved metric
- Integration: relay process – end-to-end promotion in test DB
- E2E (if applicable): admin panel shows DLQ count
```

План тестов вы пишете не для того, чтобы его подшить. Он решает, какие тесты вы напишете на
шаге 4.

### Шаг 4: сначала падающий тест

Напишите тест до реализации. Запустите его. Убедитесь, что он падает по нужной причине (фичи ещё
нет, а не из-за синтаксической ошибки или проблемы с импортом).

```ts
// Confirm this fails before the implementation exists.
it('promotes message to DLQ after 10 consecutive failures', async () => {
  const relay = createRelay(config);
  const msg = await seedMessage(db, { failureCount: 9 });

  await relay.attemptDelivery(msg.id); // 10th failure — should promote

  const dlq = await db.query('SELECT * FROM outbox_dead_letter WHERE id = $1', [msg.id]);
  expect(dlq.rows).toHaveLength(1);
});
```

Тест, который ни разу не был красным, ни разу не доказал, что он что-то ловит. Сначала заставьте
его упасть.

### Шаг 5: реализация

Напишите минимум кода, который делает падающий тест зелёным. Не добавляйте непокрытое тестами
поведение. Не выходите за границы задачи. Если реализация показывает, что спецификация неверна,
остановитесь и пройдите процесс правки спецификации, описанный в
[traceability-and-phase-reviews](/kb/process/traceability-and-phase-reviews).

Коммитьте атомарно. Каждый коммит должен выглядеть так:

```
<WI>: Short imperative description

Optional longer explanation if the change is not self-evident.
```

Примеры:
```
TASK-7: Add failure_count column to outbox table
TASK-7: Promote message to DLQ after 10 consecutive failures
TASK-7: Emit dlq.moved metric on promotion
```

Префикс с номером элемента работы — это то, что позволяет проследить любой коммит до его задачи,
стоящего за ней раздела дизайна и требований, из которых он вырос.

### Шаг 6: проверка в IDE

После каждого логического изменения проверяйте IDE на ошибки типов, нарушения линтера и
предупреждения. Не накапливайте кучу проблем IDE к концу; правьте каждую по мере появления. PR,
добавляющий новые ошибки типов или нарушения линтера, к ревью не готов.

### Шаг 7: убедитесь, что собирается

```bash
bun run build
```

Сборка должна быть чистой. PR, который не собирается, — это не PR.

### Шаг 8: проверка сначала на десктопе

Если у проекта есть десктопная цель, соберите десктопное приложение и проверьте фичу сначала там.
Полный протокол — в [desktop-target-first](/kb/process/desktop-target-first). Скриншот в качестве
доказательства обязателен.

Если проект только веб, соберите продакшен-сборку локально и проверьте в браузере через MCP, а не
только в дев-сервере.

### Шаг 9: прогон Playwright E2E

Прогоните весь набор Playwright. Всё проходит, три прогона подряд, ноль флака. Никаких
пропущенных тестов, никаких известных падений.

```bash
bun run test:e2e
```

Флакающий тест — это падающий тест. Почините его до открытия PR.

### Шаг 10: обновите документацию

Каждый PR, меняющий поведение, обязан обновить документацию:

- **README.md в каждой затронутой папке** — назначение, структура, ключевые решения.
- **documentation/user/** — гайд по фиче, если изменение видно пользователю.

Документация входит в цикл, а не прикручивается потом. PR, который меняет API без обновления
README этого модуля, незакончен.

### Шаг 11: финальная зачистка

Перед открытием PR:

- Уберите отладочный код, вызовы `console.log`, закомментированный код.
- Убедитесь, что все комментарии TODO в изменённых файлах либо закрыты, либо привязаны к
  заведённым задачам (а не «TODO: fix this later» в новом коде).
- Прогоните линтер последний раз.
- Прогоните сборку последний раз.

### Шаг 12: ручная проверка в MCP-браузере

Откройте MCP-браузер на финальной сборке и прогоните фичу ещё раз. Убедитесь, что консоль чистая.
Сделайте финальные скриншоты.

### Шаг 13: откройте PR

Описание PR следует такой структуре:

```markdown
## Summary
- What this PR delivers (one sentence per bullet, max three bullets).
- Reference to the task/issue: closes #123.

## Changes
- Brief list of implementation changes (module names, new files, removed files).

## Test plan
- What was tested and how.
- Screenshot attachments (desktop, mobile if applicable).

## Checklist
- [ ] Tests pass (all layers, zero flakes)
- [ ] Build passes
- [ ] Desktop verified (if applicable)
- [ ] Mobile verified (if applicable)
- [ ] Documentation updated
- [ ] No debug code
```

## Антипаттерны

**Начать с реализации.** Напишете код до падающего теста — и тест получится подогнанным под уже
написанный код, а не под проверку требования. Он становится зелёным на первом же прогоне и почти
ничего не говорит о покрытии.

**Свалить несколько задач в один PR.** Каждая задача в `tasks.md` соответствует одному PR.
Свалка усложняет и ревью, и бисекцию, и откат. Исключение — задачи тривиально маленькие, без
сложности в дизайне, вроде переименования или правки конфига; сгруппируйте такие с пометкой в
описании PR.

**Откладывать документацию.** Документация, написанная неделю спустя, описывает реализацию такой,
какой вы её помните, а не такой, какой вы её понимали во время сборки. Пишите её во время цикла, с
кодом перед глазами, и она выйдет точной.

**Открыть черновой PR и забыть про чек-лист.** Черновик — для раннего фидбэка по направлению. Это
не парковка для незавершённой работы с незаполненным чек-листом. Если открываете черновик,
распишите, чего не хватает и когда планируете закончить.

**Пропустить шаг сборки.** «Тесты прошли, значит, со сборкой всё в порядке.» Тесты бегут в
окружении с разрешёнными модулями, а сборка всё равно может сломаться на пути импорта, отсутствующем
ассете или конфиге бандлера. Собирайте отдельно, проверяйте отдельно.
