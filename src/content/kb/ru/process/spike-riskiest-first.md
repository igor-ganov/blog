Любая новая архитектура опирается хотя бы на одно допущение, которое никто не проверял в
этой конкретной комбинации технологий, нагрузки и окружения. Если вы построите фичи поверх
этого допущения, а оно окажется неверным, вся работа над фичами уйдёт впустую. Спайк — это
минимум работы, нужный, чтобы проверить самое рискованное допущение до начала любой работы
над фичами.

Есть и обратная ловушка — переусложнить PoC. Вы добавляете многопользовательскую обвязку,
настоящий биллинг, дашборды и шифрование ещё до того, как кто-либо проверил, оправдывает ли
себя основная ставка. Построить в PoC слишком много — это потеря такого же объёма работы,
как и пропуск спайка.

## Почему это важно

PoC сервиса на Cloudflare Workers (2026-05-22) сделал архитектурный компромисс явным. Весь
проект держался на одной технической ставке: сможет ли `mtcute` (клиентская библиотека
MTProto) работать на рантайме `workerd` от Cloudflare Workers и держать постоянную
MTProto-сессию? Если нет — выбор хостинга неверен, и всё построенное поверх рухнет вместе с
ним.

Поэтому спайк мы сделали задачей номер ноль. Не задачей номер три, не «дойдём — разберёмся».
Спайк определили конкретно:

> Worker → mtcute → собственная Telegram-сессия → прочитать N сообщений

Зелёный — архитектура валидна. Красный — немедленно менять хостинг, до того как написана
хоть строчка кода фич.

Второй урок из того же проекта — разница между дисциплиной совместимости с будущим и
инфраструктурой совместимости с будущим. PoC проектировали так, чтобы он вырос в
многопользовательскую систему, но в первый день ей быть он не обязан. Дисциплина стоит почти
ничего: протащить `userId` через сигнатуру каждой функции, с первого дня использовать ключи
хранилища с привязкой к пользователю (`user:<id>:cursor:<channel>`) и держать конвейер чистой
функцией. А вот построить саму многопользовательскую инфраструктуру (потоки аутентификации,
шифрование сессий, проверки биллинга, админ-дашборды) до того, как основной PoC проверен,
стоит недель, и всё это выброшено впустую, если спайк провалится.

## Как применять

### Определите спайк

Спайк — не прототип полной фичи. Это минимальный работающий код, который отвечает на один
конкретный вопрос, и вопрос этот всегда о том, держится ли самое рискованное допущение.

Прежде чем писать спайк, сформулируйте вопрос явно:

```markdown
Spike goal: Confirm that mtcute runs on workerd and holds an MTProto session.

Success criteria:
- A Cloudflare Worker using mtcute can authenticate with Telegram.
- The session persists across Worker invocations (via KV or Durable Object).
- Reading N messages from a channel completes without errors.

Failure criteria:
- mtcute fails to compile for the workerd target.
- The Worker hits memory or CPU limits under normal use.
- Session storage does not survive between invocations.

Time box: half a day (4 hours).
```

Тайм-бокс не обсуждается. Спайк, который тянется неделю, либо был неправильно ограничен по
объёму, либо допущение оказалось сложнее, чем ожидалось. Во втором случае сама сложность и
есть находка, и архитектурное решение нужно пересмотреть.

### Найдите самое рискованное допущение

Частые кандидаты на самое рискованное допущение:

- Библиотека, которую никогда не запускали в целевом рантайме (workerd, Deno, Bun,
  конкретный движок браузера).
- Требование по задержке или пропускной способности, зависящее от внешнего сервиса,
  который ещё не измеряли.
- API, который задокументирован, но чьё поведение на нужном масштабе или в граничном случае
  не проверяли.
- Интеграция между двумя системами, где документация по протоколу неоднозначна.

Самое рискованное допущение — то, где ошибка выбрасывает в мусор больше всего работы.
Начинайте оттуда.

### Совместимость с будущим как дисциплина

После того как спайк прошёл, не поддавайтесь желанию строить вспомогательную инфраструктуру,
которая «почти нужна». Вот как дисциплина выглядит на практике.

**Протаскивайте ось роста, но не стройте её.**

```ts
// Forward-compatible: userId is in every function signature from day one,
// hardcoded to one value. Adding multi-user later is a config change.
const fetchDigest = (config: Config, userId: UserId, cursor: Cursor) =>
  pipeline(config, userId, cursor);

// Not forward-compatible: userId is implicit or global.
// Adding multi-user later requires rewriting every function.
const fetchDigest = (config: Config, cursor: Cursor) =>
  pipeline(config, cursor);
```

**С первого дня задавайте ключам хранилища пространство имён.**

```ts
// Forward-compatible: user-scoped key, one user hardcoded.
const cursorKey = `user:${userId}:cursor:${channelId}`;

// Not forward-compatible: global key that will conflict when multi-user arrives.
const cursorKey = `cursor:${channelId}`;
```

**Держите конвейер чистым.**

```ts
// Forward-compatible: pure function, no side effects, fully testable.
const buildDigest = (
  config: DigestConfig,
  cursor: Cursor,
  messages: readonly Message[],
): { digest: Digest; newCursor: Cursor } => { ... };

// Not forward-compatible: side-effecting pipeline that mixes I/O
// with transformation, preventing isolation testing.
const buildDigest = async (config: DigestConfig) => {
  const cursor = await kv.get('cursor');
  const messages = await fetchMessages(cursor);
  await kv.put('cursor', newCursor);
  return digest;
};
```

Версии, совместимые с будущим, стоят одного лишнего параметра или одного лишнего разделителя
пространства имён. Это не преждевременная абстракция. Это наименьшее вложение, которое
избавляет от переписывания, когда продукт вырастет.

### Что PoC строить НЕ должен

Явный список из того решения по PoC (2026-05-22):

- Потоки входа других пользователей.
- Шифрование чужих Telegram-сессий.
- Настоящую логику биллинга (заглушка: `const isEntitled = () => true`).
- Админ-дашборды.
- UI для настройки.

Каждое из этого — реальная задача, и каждое получит собственную небольшую спецификацию, как
только PoC подтвердит основную ставку и направление продукта будет утверждено. Строить их до
проверки означает переусложнять прототип, который вполне может оказаться выброшенным.

## Антипаттерны

**Считать спайк прототипом.** Спайк отвечает на один вопрос. Как только вопрос закрыт, спайк
готов, даже если код сырой. Станет ли этот код продакшен-кодом — решение, которое вы
принимаете после спайка, а не во время него.

**Пропускать спайк, потому что «оно, наверное, заработает».** Весь смысл спайка в том, что вы
на самом деле не знаете, заработает ли оно. Предположить, что оно заработает, — не то же
самое, что запустить его.

**Строить инфраструктуру совместимости с будущим вместо дисциплины.** Поднять полноценную
мультиарендную систему аутентификации, «потому что она нам когда-нибудь понадобится», ещё до
того как основной PoC заработает, — это преждевременная абстракция, а не дисциплина.
Дисциплина протаскивает данные (userId в сигнатурах, ключи с пространством имён).
Инфраструктура реализует потоки (вход, управление сессиями, биллинг).

**Раздувать объём PoC по чуть-чуть.** PoC подтверждает основную ставку. Каждая лишняя задача,
которая вползает («раз уж я здесь, заодно добавлю...»), оттягивает эту проверку и повышает
шансы, что весь PoC придётся выбросить, если ставка была неверна.

## Смотрите также

Подход «сначала спайк» и дисциплина совместимости с будущим — это то, как стартует
спецификационный процесс, когда сама архитектура ещё не устоялась. Спецификация для
архитектуры, которую никто не проверил спайком, спекулятивна. Требования могут быть
достижимы, а разделы дизайна — читаться связно, но ни то ни другое не проверено против
реального рантайма. Спайк даёт доказательства, на которых последующая спецификация
становится обоснованной.
