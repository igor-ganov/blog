Эпик, который выезжает одним большим PR, тяжело ревьюить и тяжело бисектить, когда
что-то ломается, и половину его не задеплоишь. Но разбивка на инкременты не даёт
ничего, если поздние инкременты забегают вперёд или позволяют тестам краснеть. Это
всё тот же большой PR, просто переодетый в несколько штук.

Инициатива по офлайн-режиму в SPA для администрирования контента (2026-04-30) пришла
к конкретной структуре. Каждый инкремент — это одна задача в GitHub. Тело задачи
следует фиксированной схеме. Все пять слоёв тестов остаются зелёными на каждом
инкременте, без исключений. А порядок эпика строго последовательный: фаза B
заблокирована до тех пор, пока фаза A не смержена и не зелёная.

## Почему это важно

Инициатива по офлайн-режиму в SPA для администрирования контента была многофазным
выкатом возможностей. Фазы шли по порядку, потому что каждая закрепляла контракты, от
которых зависели последующие: регистрация service worker, ключи кэша, схема IDB, форма
API. Пропустите фазу A и реализуйте фазу B первой — и код фазы B сошлётся на
контракты, которых ещё нет. Тогда вы либо пишете временные заглушки, которые потом
выдерете, а это переделка, либо сворачиваете обе фазы в один большой коммит, а это
убивает ту самую инкрементальную структуру, ради которой всё затевалось.

Правило «зелёный на каждом инкременте» даёт бисектируемость. Допустим, регрессия
вылезла после фазы C. Если каждый инкремент мержился зелёным, диапазон бисекта — это
код в пределах одного инкремента. Ослабьте правило, дайте фазе B смержиться с
известными падениями «починим в фазе C» — и регрессия может прятаться в любых углах,
которые фаза B срезала.

Схема тела задачи — не украшение. Цель / Приёмка / Тесты / Вне области / Зависит от
фиксирует то, что ревьюеру действительно нужно, чтобы оценить PR: задуманный
результат, как мы понимаем, что он достигнут, что проверено, что осознанно отложено и
что должно быть уже смержено до старта.

## Как применять

### Схема тела задачи

Каждый инкремент в эпике — это задача в GitHub с ровно такой структурой тела:

```markdown
## Goal
One paragraph. What capability does this increment deliver and why does it matter
in isolation? An increment that is only useful as setup for the next one is not an
increment — it is a prerequisite that should be folded into the next phase or
extracted into a smaller, self-contained deliverable.

## Acceptance criteria
- Bullet list of observable, verifiable outcomes.
- Each criterion maps directly to a test in the Tests section.
- Written from the user or system perspective, not the implementation perspective.

## Tests
- [ ] Unit: what unit tests cover the new logic
- [ ] Integration: what integration tests cover the interaction between components
- [ ] E2E mocked: what E2E scenarios run against a mocked backend
- [ ] E2E prod: what E2E scenarios run against the real production backend
- [ ] Manual: any manual verification steps (desktop app, mobile viewport, etc.)

## Out of scope
- Explicit list of things that were considered and deliberately deferred.
- This section prevents scope creep and documents why certain related things
  are not in this increment.

## Depends on
- List of issue numbers or PRs that must be merged before this can start.
- If empty, this increment can begin immediately.
```

### Пять слоёв тестов

Все пять слоёв должны быть зелёными, когда PR инкремента мержится:

1. **Unit** — чистая логика, один модуль, без I/O.
2. **Integration** — границы компонентов, база данных, очередь сообщений или
   взаимодействия сервисов.
3. **E2E mocked** — полный поток приложения против управляемого, предсказуемого
   мок-бэкенда.
4. **E2E prod** — полный поток приложения против реального продакшен-бэкенда.
5. **Manual** — проверка человеком в реальном рантайме (скриншоты десктопного
   приложения, мобильный вьюпорт, чистая консоль).

«Все слои зелёные» означает: ни пропусков, ни флаки, ничего не отложено на следующий
инкремент. Тест, отключённый через `test.skip` или `xit`, чтобы пропихнуть мерж, — это
падающий тест. См. [никаких ретраев, никаких флаки](/kb/testing/no-retries-no-flakes).

### Порядок эпика

Нумеруйте фазы явно: `epic.1`, `epic.2`, `epic.3`. Правило:

- `epic.N+1` не может начаться, пока `epic.N` не смержен и все слои не зелёные.
- `epic.N+1` не имеет права ссылаться на контракт (API, схему, событие), который
  `epic.N` должен был закрепить, но который ещё не отревьюен и не смержен.

Если ловите себя на том, что пишете код для `epic.3`, потому что `epic.2` на ревью, а
вам хочется не простаивать, — остановитесь. Идите разбирать комментарии ревью к
`epic.2`, вместо того чтобы спекулятивно строить `epic.3` на контрактах, которые ещё
могут поменяться.

### Изоляция с ценностью MVP

Прежде чем писать задачу, спросите: если бы все последующие фазы отменили, стоил бы
этот инкремент мержа? Если ответ нет, граница проведена неверно, и тому обычно две
причины:

- Инкремент — чистая инфраструктура без видимой пользователю ценности. Посмотрите,
  нельзя ли выпустить инфраструктуру и её первого потребителя одним инкрементом.
- Инкремент — это полфичи, которая становится полезной только когда приедет следующая.
  Посмотрите, нельзя ли подкрутить область так, чтобы выдать законченную минимальную
  версию фичи уже сейчас.

«Стоит мержа сам по себе» не значит «законченный». Урезанная версия, которая работает
от начала до конца, ценна сама по себе. Коммит-каркас, набитый заглушками-реализациями,
— нет.

## Антипаттерны

**Забегание вперёд.** Фаза B стартует до того, как смержена фаза A, потому что «фаза A
по сути готова». Фаза A готова, когда она смержена и зелёная, а не когда она на ревью.
Контракты, которые она закрепляет, не стабильны, пока она не смержится.

**Откладывание падений тестов.** Тест, покрывающий поведение фазы A, падает время от
времени, и кто-то решает смержить сейчас, а починить в фазе B. Это не инкремент, это
технический долг, зашитый в набор тестов. Почините тест или почините код до мержа.

**Пустой раздел «Вне области».** Пустой раздел «Вне области» обычно означает, что
никто всерьёз не думал над границей. Любое решение по фиче что-то откладывает. Если в
голову ничего не приходит, значит, область рассмотрена недостаточно внимательно.

**Критерии приёмки, которые не проверяемы.** «Фича должна ощущаться отзывчивой» — это
не критерий. «WHEN the user clicks Save THE SYSTEM SHALL display the confirmation
within 200ms» — критерий. Критерии приёмки держат ту же планку, что и требования
EARS: проверяемы независимо.

**Цель, написанная как описание задачи.** «Реализовать модуль регистрации service
worker» — это задача. «Пользователи могут загрузить приложение в офлайн-первом режиме
после первого визита» — это цель. Раздел «Цель» описывает результат для пользователя
или системы, а не сделанную ради него работу.

## Как обеспечивается соблюдение

Описание PR ссылается на номер задачи и подтверждает, что все пять слоёв тестов
зелёные. Мерж без зелёного набора тестов — нарушение процесса, насколько бы срочной ни
была правка. Когда CI флакает, вы чините флаки до мержа, а не обходите её стороной.

Порядок эпика держится на поле «Зависит от» и на том, что PR для `epic.N+1` не
открывают, пока `epic.N` ещё не смержен. Ничто в тулинге это не навязывает, это правило
процесса. Запись здесь означает, что любое отклонение должно быть явным и нести
заявленную причину.
