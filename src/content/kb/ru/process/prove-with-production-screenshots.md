"Работает" без доказательств — это ещё не показано. Зелёные тесты локально, успешный
CI-пайплайн и работающий dev-сервер — всё это необходимо, но ни одно из них не
доказывает, что продакшен-деплой корректен. Прод работает в других условиях: ассеты
могут быть устаревшими из-за закешированного service worker, в хранилище браузера
остались привязки к ветке, CSP-заголовки отличаются, CDN кеширует по своему графику.
Каждое из этого ломало прод после того, как локально и в CI всё было зелёным, и не
один раз.

Правило не обсуждается: каждое заявление "готово" должно подкрепляться скриншотами
из прода, показанными в ответе, а не пересказанными.

## Почему это важно

Инцидент с инструментом мониторинга в продакшене (2026-04-18) — это то, откуда взялось
это правило, и взялось из реальных сбоев. Те, что и заставили его сформулировать:

1. **Устаревший service worker отдавал старые ассеты.** Новый деплой был выкачен, но
   браузеру всё ещё отвечал закешированный service worker, который продолжал тянуть
   предыдущий бандл. Локальные тесты гоняли новый код, а пользователи в проде видели
   старый. Скриншот с прод-URL, а не с localhost, поймал бы это сразу.

2. **В IndexedDB была привязана не та ветка.** Приложение хранило конфигурацию в
   IndexedDB по ключу с идентификатором ветки. Деплой на другую ветку сам по себе был
   корректен, но устаревший ключ в IndexedDB браузера от предыдущего деплоя продолжал
   его перебивать. Ни один локальный тест не трогает состояние IndexedDB в проде.

3. **CDN-кеш отдавал устаревший HTML.** HTML-оболочка кешировалась на краю сети. Деплой,
   который изменил ссылки на хеши ассетов в HTML, выкатился на origin, но CDN продолжал
   отдавать старый HTML, поэтому хеши ассетов больше не совпадали и страница не
   загружалась. Это видно в момент, когда делаешь скриншот прода, и никогда не видно в
   тестовом окружении.

Каждое из этого было заметно за секунды после открытия настоящего прод-URL в браузере и
невидимо откуда-либо ещё.

Мобильный инцидент с SPA для админки контента (2026-04-19) добавил мобильный угол: любая
мобильная работа, названная "готовой" без проверки в реальном браузере на мобильном, на
деле готова не была. Планка приёмки из того инцидента держится до сих пор. Мобильная
работа принимается только как папка со скриншотами, покрывающими каждую затронутую
страницу на мобильном вьюпорте, начиная с главной.

## Как применять

### Снятие скриншотов из прода

Наведи Chrome DevTools MCP или Playwright screenshot API на прод-URL. Не на localhost,
не на staging-превью, не на dev-сервер. Скриншот должен прийти с того же URL, который
открыл бы реальный пользователь.

```ts
// Playwright against production — the URL is the deployed origin.
const page = await context.newPage();
await page.goto('https://your-production-domain.com/path');
await page.screenshot({ path: 'proof/feature-desktop.png', fullPage: true });
```

Приложи файлы скриншотов к PR или вставь их в ответ. Покажи их, не описывай. "Страница
выглядит корректно" не доказывает ничего; скриншот доказывает.

### Покрытие мобильного вьюпорта

Для любого изменения, которое затрагивает UI, видимый на мобильном:

1. Открой Chrome DevTools на прод-URL.
2. Включи эмуляцию устройства или подключи реальное устройство.
3. Начни с главной страницы и пройди по каждой странице, затронутой изменением.
4. Сделай скриншот каждой страницы на мобильном вьюпорте.

Структура папки для прохода мобильной проверки:

```
proof/
  mobile/
    01-home.png
    02-article-list.png
    03-article-detail.png
    04-settings.png
```

Каждая страница, по порядку, начиная с главной. Приложи папку к PR. Пропущенная
страница означает, что мобильная проверка не завершена.

### Что считается продом

Прод — это окружение, к которому обращаются реальные пользователи. В порядке
предпочтения:

1. Живой прод-домен.
2. Staging-окружение, идентичное проду по всем характеристикам времени выполнения (тот
   же CDN, тот же service worker, те же env-переменные, зашитые внутрь).

Превью-деплой Vercel из ветки PR подходит для проверки до мержа, если это тот деплой,
который станет продом, то есть работает на той же инфраструктуре. Dev-сервер Vite не
считается.

### Проверка service worker

Если приложение использует service worker, проверка скриншотами из прода обязана
подтвердить, что активна правильная версия service worker:

1. Открой DevTools → Application → Service Workers.
2. Убедись, что версия service worker совпадает с задеплоенным коммитом.
3. Если активен старый service worker, нажми "Update" или очисти данные сайта и
   перезагрузи страницу.
4. Пересними скриншоты после того, как убедился, что работает правильная версия.

Скриншот из прода со всё ещё активным устаревшим service worker не доказывает ничего.

### Консоль и сеть

После визуальных скриншотов проверь консоль DevTools и панель сети:

- Нет непойманных ошибок.
- Нет упавших сетевых запросов (4xx или 5xx на ресурсах, от которых зависит
  приложение).
- Нет нарушений Content Security Policy.

Если что-то из этого всплывает, фича не готова. Она сломана в проде.

## Антипаттерны

**Пересказ скриншотов вместо их показа.** "Я проверил на мобильном, выглядит нормально"
— это не мобильная проверка. Скриншоты должны быть видны в ответе или приложены к PR.

**Использование dev-сервера как заместителя прода.** `localhost:5173` — это не прод.
Service worker'ы, кеширование CDN и зашивание env-переменных там никогда не работают.

**Проверка только изменённой страницы.** Изменение в общем компоненте или глобальном
стиле может сломать страницы, которые никто специально не тестировал. Именно поэтому
папка мобильной проверки начинается с главной страницы.

**Заявление, что скриншот из CI — это скриншот из прода.** CI снимает скриншот браузера
Playwright против dev-сборки или превью-URL. Пока этот URL не является настоящим
продакшен-деплоем с теми же характеристиками времени выполнения, он не считается
доказательством из прода.

**Пропуск проверки для "тривиальных" изменений.** Сбой с устаревшим service worker
пришёл из того, что выглядело тривиальным изменением ассета. Нет порога, ниже которого
можно пропустить проверку в проде.

## Принуждение к исполнению

Определение готовности для доставки: PR завершён, когда содержит скриншоты из прода
(десктоп и мобильный, если задействован UI), приложенные или встроенные в описание PR.
PR без скриншотов для UI-изменения не готов к мержу.

Постоянный приоритет: если тебе сказано подключиться к экземпляру Chrome или открыть
браузер через токен, а ты игнорируешь это в пользу локального теста — работа не сделана.
Просьбу использовать конкретный браузер или окружение нужно выполнять так, как она
дана.
