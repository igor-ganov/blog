Последовательный E2E-набор медленный, и эта медлительность делает работу, которую вы
от неё не ждёте. Каждый тест получает всю машину, страницы оседают неспешно, а гонки
прячутся в этом запасе времени. Поднимите число воркеров — и набор станет заметно
быстрее, а каждый тест, который проходил по случайности, начнёт честно падать. Вам
нужно и то, и другое.

Правило: CI прогоняет набор с параллельными воркерами, и любое падение, спровоцированное
параллелизмом, — это настоящий дефект приложения или сигнала ожидания. Уменьшать число
воркеров, чтобы «стабилизировать» набор, делает то же, что и добавление ретраев: это
покупает зелёный, убирая нагрузку, которая обнажает гонку.

## Почему это важно

В SPA админки контента (2026-06-12) пайплайн деплоя гонял E2E с `workers: 1` в CI.
Подъём до 4 сократил шаг E2E с 6m55s до 2m46s — самая большая экономия в 9.5-минутном
пайплайне. Заодно это сломало три теста за первые несколько прогонов. Ни один из трёх
не был проблемой теста:

1. Навигация после логина время от времени умирала с `net::ERR_ABORTED` — свежая
   активация service worker перехватывает клиент прямо во время `goto`. Багу не один
   месяц; последовательные прогоны были достаточно медленными, чтобы активация всегда
   выигрывала гонку.
2. Переключение раздела проверяло элементы списка до того, как пришли данные раздела —
   данные идут через `MessageChannel`, невидимый для ожиданий по сети. См.
   [внеполосному транспорту нужны DOM-сигналы](/principles/testing/out-of-band-transport-needs-dom-signals).
3. Визиты опирались на `networkidle`, который на деле скрытый sleep на 500ms и ничего
   не доказывает о жизненном цикле SW. См.
   [дождитесь, пока service worker устаканится](/principles/testing/wait-for-service-worker-settle).

Каждая правка была событийной по форме и делала поведение приложения честнее. После
этого набор прошёл 271/271 три раза подряд локально и оставался зелёным в CI — быстрее
прежнего и достойным доверия.

## Как применять

**Шаг 1: параллельность в CI, явно.**

```ts
// playwright.config.ts
export default defineConfig({
  // CI runners have ~4 vCPUs; saturate them. Locally let Playwright decide.
  workers: process.env.CI ? 4 : undefined,
});
```

**Шаг 2: одна переменная окружения для потолка ожидания — никаких прокладок в спеках.**

Четыре воркера на четырёх общих vCPU делают здоровый холодный старт честно медленнее,
чем на выделенной локальной машине. Это меняет *потолок*, до которого ожидание может
блокироваться перед падением, а не само ожидание. Сделайте потолок переопределяемым
через окружение в общих хелперах ожидания, а строгое значение по умолчанию оставьте
локально:

```ts
// wait helper (shared toolkit)
const envMax = Number(process.env.E2E_MAX_WAIT_MS);
const maxMs = Number.isFinite(envMax) && envMax > 0 ? envMax : 10_000;
```

```yaml
# deploy.yml — CI is slower, not buggier; say it once.
env:
  E2E_MAX_WAIT_MS: '30000'
```

Ожидания остаются событийными и разрешаются в тот же миг, как условие становится
истинным; двигается только дедлайн падения. Когда случайные хвостовые тесты начинают
выпадать по таймауту в CI, поднимайте эту одну переменную. Не рассыпайте
`{ timeout: 30000 }` по отдельным спекам — каждый такой случай прячет, спек медленный
или сломанный.

**Шаг 3: проверьте, что последовательный набор тихо не запускал.**

Работа над скоростью заставляет вас перечитать набор, и найденное там может значить
не меньше, чем тайминги. Тест, загружающий PDF, указывал фикстурой на **абсолютный
путь на машине автора** под защитой `test.skip`, когда файла нет. Он месяцами тихо
пропускался в CI, пока набор рапортовал зелёный. Фикстурам место в репозитории
(`e2e/fixtures/`), а условный skip на отсутствующую фикстуру — это
[пропущенный тест, то есть упавший тест](/principles/testing/no-retries-no-flakes).

## Антипаттерны

```ts
// ❌ Stabilising by de-parallelising. The race is still in production.
workers: 1,

// ❌ fullyParallel: false / serial mode for a flaky describe block —
//    same move, smaller blast radius, same hidden race.
test.describe.configure({ mode: 'serial' });

// ❌ Per-spec timeout padding. Now nobody knows the real budget.
await expect(item).toBeVisible({ timeout: 45_000 });

// ❌ Fixture outside the repo, guarded by a skip — green CI, zero coverage.
const PDF = 'C:/Users/author/Downloads/sample.pdf';
test.skip(!fs.existsSync(PDF), 'fixture missing');
```

## Контроль

Бо́льшую часть этого обеспечивает сам пайплайн: параллельные воркеры в CI, ноль
ретраев и правило трёх прогонов. Остальное уходит в ревью. Проверяйте, что `workers`
не равен 1 в CI, что `mode: 'serial'` не появляется без письменного обоснования, что
в спеках не живут литеральные переопределения таймаутов (потолку место в общем
хелпере) и что каждый путь к фикстуре разрешается внутри репозитория.
