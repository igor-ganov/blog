Ролевые локаторы Playwright видят доступные имена через сопоставление по подстроке.
`getByRole('button', { name: /new/i })` находит каждый интерактивный элемент, чьё
доступное имя содержит "new", в том числе те, в которые вы никогда не целились. Поэтому
когда какой-то посторонний компонент получает aria-label, случайно содержащий слово из
вашего тестового словаря, локатор начинает совпадать сразу с несколькими элементами, и
тест умирает с сообщением "strict mode violation: locator resolved to 2 elements".

Чините это на уровне ярлыка. Ослабление локатора лишь прячет настоящую причину —
коллизию имён.

## Почему это важно

В SPA для администрирования контента (2026-04-30) индикатор уведомлений получил доступное
имя `"No new notifications"`. Точное, описательное, само по себе совершенно нормальное.
А потом в одном прогоне CI упало одиннадцать наборов тестов на Chromium. На поиск причины
ушло двадцать минут: `getByRole('button', { name: /new/i })`, которым по всему набору
целились в кнопку **"Create new content"**, теперь совпадал с двумя элементами — кнопкой
создания и индикатором уведомлений. Оба сидели в раскладке страницы, оба имели роль
`button`, и оба несли подстроку `new`.

Переименование ярлыка уведомлений в `"Notifications: none unread"` всё починило — имя с
существительным впереди и без зарезервированного словаря. Все одиннадцать наборов
восстановились.

Масштаб поломки и делает принцип осязаемым. Одно изменение ярлыка на компоненте, который
рендерится на каждой странице, стоило больше часа отладки по всему набору среднего
размера. Зарезервированный словарь здесь такой:

- `new`, `create`, `save`, `delete`, `add`, `remove`, `edit`

Любой aria-label на индикаторе статуса, бейдже, счётчике или декоративном элементе,
содержащий одно из этих слов, рано или поздно столкнётся с ролевым локатором, который
целится в действие. В большом приложении коллизия может проявиться спустя месяцы после
того, как ярлык был написан, — в тот день, когда несвязанная команда добавит конфликтующий
тест.

## Как применять

**Используйте имена с существительным впереди и двоеточием для статусов и индикаторов.**

Шаблон с двоеточием (`Существительное: значение`) — устоявшееся соглашение для доступных
имён статусных областей. Оно отделяет роль элемента от его текущего состояния, естественно
читается, когда его озвучивает экранный диктор ("Notifications двоеточие none unread"), и
держит словарь действий вне ярлыка.

```ts
// ❌ Dangerous — contains "new", matches /new/i role locators
html`<button aria-label="No new notifications" ...>`

// ❌ Also dangerous — "delete" matches /delete/i
html`<span aria-label="No items to delete" ...>`

// ✅ Noun-led, colon-separated, no action vocabulary
html`<button aria-label="Notifications: none unread" ...>`
html`<button aria-label="Sync status: idle" ...>`
html`<span aria-label="Queue: 0 items" ...>`
```

Прежде чем остановиться на каком-либо aria-label, прогрепайте набор тестов на ролевые
локаторы, которые с ним совпали бы:

```
grep -r "getByRole.*name.*new" e2e/
grep -r "getByRole.*name.*create" e2e/
```

Если grep находит совпадение в постороннем тесте, предложенный ярлык конфликтует и должен
быть изменён.

**Предпочитайте `data-testid` для основной навигации в тестах; оставьте ролевые локаторы
для проверок доступности.**

Ролевые локаторы — правильный инструмент, чтобы утверждать, что до элемента можно
добраться по его роли и имени, ведь именно это они и проверяют. Но как основная точка
навигации они хрупки, потому что доступные имена — это видимый пользователю текст, который
перелопачивают переводы, правки копирайтеров и A/B-тесты. Привязывайте навигацию к
`data-testid` из расположенной рядом константы-локатора, а ролевой локатор пусть будет
вторичной проверкой, подтверждающей, что доступное имя — именно такое, как вы ожидаете.

```ts
// ❌ Role locator as primary navigation — fragile against copy changes and collisions
await page.getByRole('button', { name: /create/i }).click();

// ✅ testid for navigation, role locator for the accessibility assertion
await page.getByTestId(TOOLBAR.createButton).click();
await expect(page.getByTestId(TOOLBAR.createButton)).toHaveAccessibleName(
  'Create new content',
);
```

Такое разделение ещё и делает намерение очевидным. Клик ведёт к фиче, а ролевая проверка
удостоверяет, что доступное имя присутствует и корректно.

## Анти-паттерны

```ts
// ❌ Substring role locator as the sole locator — brittle against label changes
//    and will match any future element whose name contains the substring.
const btn = page.getByRole('button', { name: /new/i });
await btn.click(); // breaks when notifications badge gets "No new notifications"

// ❌ Fixing the collision by making the locator more specific in the test,
//    rather than fixing the label.
const btn = page.getByRole('button', { name: /^Create new content$/i });
// This works today, but the label is still a trap for the next team member
// who writes a new /new/i locator and hits the same collision.

// ❌ Status text that mirrors action vocabulary without the noun-led structure.
html`<p aria-live="polite">Ready to save ${count} items</p>`
// matches /save/i — collides with save-button locators throughout the suite
```

Все три делают одну ошибку: считают доступное имя приватным для своего компонента. Это не
так. Любой вызов `getByRole` в Playwright где угодно в наборе тестов видит это имя — на
любой странице, где рендерится компонент, так что доступные имена — это фактически общее
пространство имён.

## Контроль

Прежде чем мёрджить компонент, который вводит новый интерактивный элемент:

1. Прочитайте предложенный aria-label.
2. Выпишите каждое значимое слово.
3. Прогоните `grep -r "getByRole.*name.*<word>" e2e/` для каждого слова.
4. Если хоть один grep вернул совпадение, поменяйте ярлык на вариант с существительным и
   двоеточием.

Это занимает тридцать секунд и предотвращает класс отказов, отладка которых стоит двадцать
минут каждый раз, когда они всплывают. Если набор тестов достаточно велик, чтобы это
оправдать, автоматизируйте проверку отдельным lint-правилом на строки aria-label.

И ещё одно. Когда пишете новый ролевой локатор для кнопки действия, делайте регулярное
выражение или точную строку настолько специфичными, насколько можете. `{ name: 'Create
new content' }` (точно, с учётом регистра) куда менее склонно к коллизиям, чем
`{ name: /new/i }`. Оставьте сопоставление по регулярному выражению для содержимого,
которое действительно меняется: счётчиков, дат, пользовательского текста.
