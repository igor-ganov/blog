`waitForTimeout(500)` в тесте признаёт одно из двух. Либо вы не знаете, чего именно
ждёте, либо приложение недостаточно быстрое и детерминированное, чтобы дождаться
настоящего сигнала. И то и другое — баги, один в тесте, другой в приложении, а таймаут
маскирует тот, что у вас есть. Он превращает зелёный прогон в, скорее всего, зелёный
и гарантирует моргание на самой медленной машине CI.

Поэтому правило такое: тесты синхронизируются по событиям, то есть по событиям DOM
и сети, и больше ни по чему. Таймауты ожидания событий (максимум, который ожидание
будет блокировать до провала) остаются минимальными, потому что корректное приложение
выдаёт событие быстро. Если нет — вы чините приложение, а не ожидание.

## Почему это важно

Стандарт здесь появился раньше набора тестов и не гнётся: **приложение должно
открываться и отвечать менее чем за одну секунду, без исключений** (десктопный
UI-инструмент, 2026-03-12). Всё остальное вытекает отсюда. Никаких таймаутов по простою
как сигнала завершения, никаких повторов теста, а порог приёмки такой: **прогоните тесты
три раза, и если хоть один прогон упал — код сломан и переписывается.** Таймауты и повторы —
костыли, которые маскируют сломанный код.

Вот конкретный провал, который привил эту дисциплину. На админ-панели свежий тестовый
`BrowserContext` гонится с циклом сервис-воркера install → activate → `controllerchange`
→ `location.reload()`. Тест, который ждал фиксированное время или просто
`domcontentloaded`, кликал по элементу на DOM, который вот-вот выбросят, перезагрузка
уводила со страницы, и клик падал с «navigated to /». Прерывисто, зависело от платформы,
невидимо, пока не доходило до CI. Мы не чинили это более длинным таймаутом. Мы ждали
настоящего сигнала готовности: предиката по состоянию жизненного цикла SW плюс стабильного
якорного элемента. См. [дождитесь готовности сервис-воркера](/principles/testing/wait-for-service-worker-settle).

Есть ловушка второго порядка, которую мы нашли позже (2026-06-12). **`networkidle` —
это замаскированный таймаут.** Он срабатывает через 500 мс сетевой тишины, так что
каждый вызов платит фиксированные полсекунды, даже если страница успокоилась мгновенно,
и страница может быть в сетевом простое, пока то, что вам на самом деле нужно, всё ещё
в полёте. То же касается любого состояния загрузки «idle», используемого как сигнал
завершения. Ждите само состояние, а не тишину.

## Как применять

Ждите того, что действительно говорит о готовности.

```ts
// ❌ Guessing how long the request takes.
await page.click('[data-testid="save"]');
await page.waitForTimeout(1000);
await expect(page.getByText('Saved')).toBeVisible();

// ✅ Wait on the network response and the DOM that proves it landed.
const saved = page.waitForResponse(
  (res) => res.url().endsWith('/tickets') && res.request().method() === 'POST',
);
await page.getByTestId(SAVE_BUTTON).click();
await saved;
await expect(page.getByTestId(SAVE_CONFIRMATION)).toBeVisible();
```

Для состояния приложения, которое разрешается через несколько асинхронных шагов, ждите
видимого пользователю следствия (область статуса переключается в «idle» или появляется
строка), используя авто-повторяющиеся проверки Playwright вроде `toBeVisible`
и `toHaveText`. Они опрашивают DOM и разрешаются в тот же миг, когда условие выполняется,
поэтому реагируют на событие, а не на часы.

Когда тест действительно моргает, план действий следственный, а не косметический:

1. Запустите сервер, поуправляйте им через Chrome DevTools / Playwright MCP, воспроизведите
   действие вручную и смотрите на консоль и сеть. Включите троттлинг.
2. Если под троттлингом всё работает и провал не воспроизводится — тест ждал не того
   события. Перепишите его, чтобы он срабатывал по другому, правильному событию DOM.
   **Не по таймауту.**
3. Если он действительно нестабилен в каких-то сценариях — в приложении гонка. Чините
   первопричину. Если архитектура не может гарантировать детерминированное поведение,
   архитектура неверна — переделайте её.

## Антипаттерны

```ts
// ❌ Idle timeout as a completion mechanism. Sentinel detection must be instant
//    and deterministic, not "probably done after 800ms".
await sleep(800);

// ❌ Retrying until it passes. A test that needs retries is reporting a real race;
//    retries hide it.
test.describe.configure({ retries: 3 });

// ❌ Browser-specific timing hacks. If a test needs a hack for WebKit, the app
//    behaves differently on WebKit and that is the bug.
if (browserName === 'webkit') await page.waitForTimeout(300);
```

## Принуждение

Прогоняйте наборы с `--reporter=list` во время разработки и с `--reporter=json`, чтобы
читать трассировки, когда что-то нестабильно. Программное исключение тестов запрещено —
во время разработки можно прогонять подмножество, но единственное определение зелёного —
это **полный, стабильный проход без единого повтора, три прогона подряд.** Моргающий или
пропущенный тест — это упавший тест.

## Смотрите также

Всё это сводится к одному предпочтению: детерминированные системы вместо вероятностных
костылей. Тот же стандарт (отклик меньше секунды, никаких таймаутов по простою, никаких
повторов, прогон три раза) явно формулирует [никаких повторов, никаких морганий](/principles/testing/no-retries-no-flakes).
