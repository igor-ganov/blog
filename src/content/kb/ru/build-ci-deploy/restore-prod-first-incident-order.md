Когда продакшен сломан, восстанови работу сервиса. Разобраться, почему он сломался, написать
тест, доказывающий поломку, открыть PR с чистым архитектурным решением — всё это нужно, но
всё это после того, как сайт снова зелёный.

Желание «сделать всё как надо» прямо во время простоя понятно: так не приходится деплоить
хотфикс, который тут же перекроется настоящим решением. Но на практике это держит сайт лежащим
дольше, пока ты пишешь тесты и ждёшь ревью, а пользователи и заказчики смотрят, как простой
тянется. Восстановление в два шага (сейчас хотфикс, потом нормальное решение) почти всегда
обходится дешевле, чем восстановление в один шаг, которое занимает втрое дольше.

## Почему это важно

**SPA для администрирования контента, 2026-05-05.**

Сборка публичного сайта упала на ошибке разбора YAML в контентном файле
(корневую причину см. в [no-self-rolled-yaml](/kb/error-handling/no-self-rolled-yaml)).
Лог сборки указывал на проблемный файл. Правильная последовательность была такой:

1. Найти сломанный файл по логу CI.
2. Залатать контент прямо в контентном репозитории (не в репозитории приложения — там
   `src/content` в gitignore; контент живёт в соседнем репозитории).
3. Запушить патч. Автоматизация создаёт пустой коммит в репозитории приложения, что запускает
   передеплой.
4. Дождаться, пока деплой станет зелёным. Убедиться, что сайт поднялся.
5. И только потом: открыть PR (или несколько), устраняющие корневую причину и добавляющие тесты.

Первым побуждением команды было начать с шага 5: открыть «чистый» PR с правильным использованием
YAML-библиотеки, добавить регрессионные тесты, провести ревью, смержить, задеплоить. Всё это
время сайт оставался красным, и каждая минута простоя, потраченная на код-ревью и написание
тестов, была лишней.

Сам хотфикс — заключить враждебную YAML-строку в кавычки в контентном файле — занял меньше двух
минут, как только сломанный файл был найден. Разрыв между «нашли» и «сайт зелёный» должен был
составить меньше пяти минут. Не составил.

## Как применять

### Шаг 1: Определить сломанное состояние

```sh
# Get the failing run ID from the most recent workflow run
gh run list --workflow deploy.yml --limit 5

# View the log for the failing steps only
gh run view <run-id> --log-failed
```

Флаг `--log-failed` фильтрует только упавшие шаги — при ошибке сборки он сразу указывает на
файл и саму ошибку. Не читай весь лог, читай только место падения.

### Шаг 2: Залатать в правильном репозитории

Если падение во входных данных сборки (контент, конфиг, переменные окружения), а не в коде
приложения, патч идёт в репозиторий с входными данными, а не в репозиторий приложения.

Для контентного пайплайна этого SPA:

```sh
# Content lives in the content repo, not in the application repo
# src/content is listed in the application repo's .gitignore

# In the content repo:
git checkout master
# Edit the broken file — quote the hostile YAML string
# title: 'Correct: with colon' rather than title: Correct: with colon
git add content/articles/the-broken-file.md
git commit -m "hotfix: quote colon in title to fix build"
git push
# Automation triggers an empty commit on the application repo, which triggers redeploy
```

### Шаг 3: Убедиться, что деплой зелёный

Не переходи к шагу 4, пока деплой не подтверждён как зелёный. Смотри прогон воркфлоу вживую
или опрашивай его:

```sh
gh run watch --workflow deploy.yml
# or
gh run list --workflow deploy.yml --limit 1
```

Сайт должен быть поднят и корректно отдавать контент до того, как начнётся фаза разбора
полётов. Если исправление визуальное, сделай скриншот продакшена (см.
[prove-with-production-screenshots](/kb/process/prove-with-production-screenshots)).

### Шаг 4: Открыть PR с устранением причины

После того как сайт зелёный, открой PR, который:

- Чинит лежащий в основе код, который вообще допустил поломку (например, заменяет
  самописный сериализатор YAML на настоящую библиотеку).
- Добавляет регрессионный тест, который поймал бы поломку до того, как она дошла до CI.
- Описывает инцидент в описании PR с таймлайном и корневой причиной.

Ревьюь этот PR внимательно и мержи через обычный процесс без спешки — сайт уже зелёный.

### Враждебные YAML-строки — конкретный шаблон исправления

Инцидент 2026-05-05 был вызван значением YAML с двоеточием. Немедленный хотфикс — взять строку
в кавычки:

```yaml
# ❌ Broken — colon after space is a YAML mapping indicator
title: An article about REST: designing APIs

# ✅ Fixed — single-quoted string; YAML allows colons inside single quotes
title: 'An article about REST: designing APIs'

# ✅ Also valid — double-quoted
title: "An article about REST: designing APIs"
```

Файлы контентного репозитория редактируются руками, так что человек ставит кавычки меньше чем
за минуту. Нормальное решение (YAML-библиотека в сериализаторе, проверка разбора на pre-commit)
идёт в следующем PR.

### Запуск передеплоя без изменения кода

Некоторым проектам нужно изменение кода в репозитории приложения, чтобы запустить CI, даже
когда исправление лежит в другом репозитории. Пустой коммит решает задачу:

```sh
# In the application repo
git commit --allow-empty -m "chore: trigger redeploy after content hotfix"
git push
```

Как вариант — используй триггер `workflow_dispatch` в GitHub Actions, если воркфлоу его
поддерживает:

```sh
gh workflow run deploy.yml --ref main
```

## Антипаттерны

Все они затягивают простой без всякой причины:

**Открывать «чистый» PR, пока сайт лежит.** PR нужно ревью, ревью занимает время, и всё это
время сайт лежит. Хотфикс восстановил бы сервис за минуты.

**Сначала писать регрессионные тесты.** Тесты подтверждают, что баг существует, — но это уже
подтверждает сам факт простоя продакшена. Тесты нужны, но их место в PR с устранением причины,
а не в хотфиксе.

**Пытаться полностью понять корневую причину до того, как действовать.** «Хочу понять, почему
это произошло, прежде чем что-то трогать» — правильный настрой для разбора полётов и неправильный,
когда сайт красный. Сначала восстанови, потом разбирайся.

**Латать не тот репозиторий.** Если контент живёт вне репозитория приложения, хотфикс идёт в
контентный репозиторий. Патч не в тот репозиторий запускает деплой, который не несёт исправления.

```sh
# ❌ Wrong repo — the content site's src/content is gitignored; this change has no effect
cd content-repo
vim src/content/articles/the-broken-file.md
git add src/content/articles/the-broken-file.md
git commit -m "hotfix"
git push
# The broken file is sourced from the content repo at build time.
# This commit changes a gitignored file; the build is unchanged.
```

## Как закрепить

Это правило процесса, а не кода. Закрепляй его через:

1. **Ранбук инцидентов.** Письменный ранбук в `docs/` репозитория, где прописаны эти четыре
   шага. Новые члены команды читают его при онбординге, и он остаётся авторитетным источником
   по порядку действий при инциденте.

2. **Разбор после инцидента.** После каждого инцидента на продакшене краткий письменный разбор
   фиксирует таймлайн, корневую причину и то, был ли соблюдён порядок действий. Отмечай
   отклонения без поиска виноватых и позволяй закономерности отклонений направлять обновления
   ранбука.

3. **Готовность дежурного.** Ещё до того, как случится инцидент, дежурный инженер должен знать,
   какой репозиторий содержит контент, какой репозиторий запускает деплои и как запустить
   передеплой без изменения кода. Выяснять это во время простоя уже поздно.
