«Деплой медленный» ничего не говорит о том, куда уходит время; таблица
длительностей по шагам говорит. CI-провайдеры отдают время каждого шага через API,
так что вытащи его прежде, чем что-то трогать. Интуиция о том, куда уходит время
пайплайна, ошибается надёжно, а сэкономленные 30 секунд на 30-секундном шаге не
видны внутри десятиминутного прогона.

## Почему это важно

Десятиминутный деплой в SPA админки контента (2026-06-12) раскладывался так:
юнит-тесты 70s, сборка 29s, установка браузеров 25s, E2E 6m55s, хвост деплоя
30s. Один шаг занимал 70% пайплайна, и ничего другого не имело смысла трогать,
пока он не уменьшится. Распараллеливание E2E-воркеров сократило шаг до 2m46s, а
пайплайн до 5m36s. Мелкие рычаги (кеширование, ограничения установки) стали
стоить усилий только после того, как сработал крупный.

Две побочные поломки во время той же работы научили операционной половине
правила:

- Шаг `playwright install` дважды висел больше часа, сжигая минуты раннера и
  блокируя очередь, потому что Node 24 в PATH ломал установщик. Без
  `timeout-minutes` регрессия инструмента на этом шаге превращается в часовое
  молчаливое зависание. Ограничь его — и та же регрессия покажется красным
  крестиком за минуты.
- Починкой стала фиксация Node 22. Эта фиксация выглядит произвольной, если
  workflow не объясняет, зачем она здесь. Необъяснённые фиксации «подчищает»
  следующий рефакторинг, и зависание возвращается.

## Как применять

**Шаг 1: измерь.** Вытащи длительности шагов из API, а не пролистывая логи:

```sh
gh run view <run-id> --json jobs \
  --jq '.jobs[].steps[] | {name, startedAt, completedAt}'
```

Отсортируй по длительности, по убыванию, и трать усилия строго сверху вниз.
[Дисциплина «сначала самое рискованное/крупное»](/principles/process/spike-riskiest-first)
работает и для пайплайнов.

**Шаг 2: кешируй загрузки по версии инструмента.**

```yaml
- name: Get Playwright version
  id: pw
  run: echo "version=$(bun pm ls | grep @playwright/test | …)" >> "$GITHUB_OUTPUT"
- uses: actions/cache@<sha>
  with:
    path: ~/.cache/ms-playwright
    key: playwright-${{ runner.os }}-${{ steps.pw.outputs.version }}
```

Ключ по версии, а не по хешу лок-файла, означает, что кеш переживает несвязанные
обновления зависимостей и инвалидируется ровно тогда, когда меняются браузеры.

**Шаг 3: ограничь каждый шаг установки.**

```yaml
- name: Install Playwright browsers
  timeout-minutes: 8   # a healthy install takes ~25s; a hang is a regression
  run: bunx playwright install --with-deps
```

**Шаг 4: фиксируй то, что сломалось, и пиши прямо здесь, почему.**

```yaml
- uses: actions/setup-node@<sha>
  with:
    node-version: 22  # Node 24 makes `playwright install --with-deps` hang (2026-06-12)
```

## Антипаттерны

```yaml
# ❌ Tuning the 30s step while a 7-minute step sits untouched — no measurement.

# ❌ Unbounded install. A registry hiccup or tool regression = 1h of runner time.
- run: bunx playwright install --with-deps   # no timeout-minutes

# ❌ Cache keyed on the lockfile — invalidates on every unrelated dep bump.
key: playwright-${{ hashFiles('bun.lock') }}

# ❌ A bare pin with no reason. The next cleanup PR unpins it and the
#    hour-long hang returns with no paper trail.
node-version: 22
```

## Как обеспечить

После любого изменения пайплайна снова вытащи таблицу длительностей и впиши в PR
оба числа (до/после). Что проверять на ревью: у каждого шага установки или
загрузки есть `timeout-minutes`, каждый ключ кеша кодирует версию инструмента,
который кеширует, и каждая фиксация версии несёт причину прямо в файле с датой.
