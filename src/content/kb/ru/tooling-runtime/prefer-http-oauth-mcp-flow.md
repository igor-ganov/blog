## Почему это важно

31.05.2026, настраивая интеграцию MCP для Microsoft Teams / M365, я сравнил
два пакета MCP-серверов:

- `@floriscornel/teams-mcp` — stdio-сервер, который авторизуется запуском собственной
  CLI-команды входа. Авторизоваться приходится через механизм самого пакета, снаружи
  MCP-клиента.
- `@softeria/ms-365-mcp-server` — HTTP-сервер, который отдаёт эндпойнт `/mcp` с кнопкой
  входа. Авторизация проходит внутри собственного OAuth-потока MCP-клиента.

Первый я отклонил. Проблема тут структурная. stdio-сервер MCP, который сам занимается
авторизацией, никак не может показать кнопку входа внутри MCP-клиента, поэтому ассистент
не пройдёт авторизацию за вас. Вы запускаете отдельную CLI-команду, состояние авторизации
живёт вне MCP-сессии, и любое обновление токена или повторный вход снова выдёргивают вас
из MCP-клиента, чтобы проделать всё заново.

Второй пакет работает потому, что авторизацией управляет клиент. MCP-клиент открывает
эндпойнт `/mcp`, сервер показывает кнопку входа, вы нажимаете её и проходите OAuth-согласие
в браузере, после чего токен сессии сохраняется. Всё это остаётся внутри собственного
интерфейса MCP-клиента.

Это обобщается на любой MCP-сервер, который вы оцениваете: предпочитайте тот, что
авторизуется через `/mcp` (HTTP + OAuth), а не через CLI-команду.

## Как применять

### Настройка MCP-сервера на HTTP+OAuth (на примере Teams)

Установите и запустите сервер локально:

```bash
# Run the HTTP MCP server on port 8765 with org-mode and auth tools enabled
bunx @softeria/ms-365-mcp-server --http 8765 --org-mode --enable-auth-tools
```

Зарегистрируйте его в MCP-клиенте как пользовательский сервер, указывающий на HTTP-эндпойнт:

```json
{
  "mcpServers": {
    "teams": {
      "url": "http://localhost:8765/mcp"
    }
  }
}
```

При первом подключении клиент запрашивает `/mcp`, а сервер возвращает кнопку входа.
Вы проходите OAuth в браузере, и сессия сохраняется, переживая перезапуски MCP-клиента.

### Автозапуск локального HTTP-сервера в Windows без прав администратора

Запланированной задаче нужны права администратора. Чтобы их избежать, положите элемент
автозагрузки в пользовательскую папку Startup через VBS-обёртку, которая запускает процесс
скрыто, без окна консоли:

```vbscript
' start-ms365-mcp.vbs — place in shell:startup
Set WshShell = CreateObject("WScript.Shell")
WshShell.Run "cmd /c bunx @softeria/ms-365-mcp-server --http 8765 --org-mode --enable-auth-tools", 0, False
```

Поместите файл `.vbs` сюда:
`C:\Users\<your-username>\AppData\Roaming\Microsoft\Windows\Start Menu\Programs\Startup`

Теперь сервер запускается при входе в систему, скрыто и без прав администратора.

### Оценка нового пакета MCP-сервера по типу авторизации

Прежде чем брать любой MCP-сервер, ответьте на эти вопросы:

1. Стартует ли он как HTTP-сервер с эндпойнтом `/mcp`? Если да — он поддерживает поток
   с кнопкой входа.
2. Авторизуется ли он через CLI-команду (`mcp-server login`, `mcp auth` и т. п.)? Если да,
   авторизация лежит вне MCP-клиента, так что лучше взять альтернативу.
3. Если есть только stdio-пакет — нет ли локальной HTTP-обёртки или прокси, отдающего
   `/mcp`? Некоторые пакеты поддерживают оба режима через флаг `--http`.

```bash
# Check if a package supports HTTP mode
bunx <package-name> --help | grep -E "\-\-http|\-\-port|http"
```

### Блокировка на согласии тенанта — не проблема пакета

В той же сессии переход с `@floriscornel/teams-mcp` на
`@softeria/ms-365-mcp-server` ничего не дал для экрана «Need admin approval» во время
OAuth-согласия. Этот экран — следствие политики тенанта: в целевом тенанте отключено
самостоятельное согласие на приложения, поэтому согласие приложению должен выдать
администратор Azure AD.

К выбору пакета это отношения не имеет. Мы выбрали правильный, а блокировка всё равно была
внешней. Не возвращайтесь к stdio-пакету в надежде, что он обойдёт политику тенанта, —
он не может.

## Антипаттерны

### Выбор stdio-сервера потому, что он первый в выдаче поиска

```bash
# Bad — installs a stdio server with CLI-driven auth
bun add -g @floriscornel/teams-mcp
teams-mcp login  # auth outside MCP client; fragile
```

Симптом: MCP-клиент показывает сервер как подключённый, но вызовы инструментов падают
с «not authenticated», потому что токен сессии из CLI-входа так и не пробрасывается
в контекст MCP-сессии.

### Допущение, что HTTP-сервер на порту означает поддержку /mcp

Некоторые HTTP-серверы отдают обычный REST API и просто слушают порт, вовсе не реализуя
HTTP-транспорт MCP. Проверьте это, запросив `/mcp` напрямую:

```bash
curl -i http://localhost:8765/mcp
# A valid MCP HTTP server returns 200 or a redirect with MCP protocol headers
# A non-MCP HTTP server returns 404 or an unrelated response
```

### Несохранение состояния авторизации между перезапусками MCP-клиента

MCP-серверы на HTTP+OAuth хранят токен сессии в памяти или на диске — зависит от
реализации. Если сервер держит его только в памяти и перезапускается при каждом старте,
вы авторизуетесь заново каждую сессию. Берите реализации, которые сохраняют токен в
локальный файл или системное хранилище ключей либо используют долгоживущие refresh-токены.

```bash
# Check if the server process has persistent token storage
# Look for a --token-store, --persist, or --data-dir flag
bunx @softeria/ms-365-mcp-server --help | grep -E "token|persist|store|data"
```

## Принуждение

Это критерий выбора MCP-сервера, а не автоматическое правило линтера.
Когда кто-то предлагает новую интеграцию MCP, требуйте задокументировать, какой поток
авторизации использует пакет, до того как его примут. Если пакет работает только через
stdio с авторизацией через CLI и без HTTP-режима, запишите решение и тот компромисс,
на который вы идёте, до слияния.
