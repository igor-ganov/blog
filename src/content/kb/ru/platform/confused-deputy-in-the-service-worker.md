Service Worker в роли backend-for-frontend в итоге держит у себя токен пользователя;
это его работа. Маршруты внутри него затем выполняют вызовы
GitHub API с этим токеном. SW обслуживает
*каждый* скрипт, работающий на этом origin, а не только UI-компоненты,
которые написали вы. Привилегированный маршрут, доверяющий вызывающей стороне, может
быть запущен любым скриптом с того же origin — будь то плацдарм XSS,
скомпрометированная зависимость или расширение браузера с доступом к странице.

Это и есть классический сбитый с толку посредник: компонент с полномочиями (токен)
выполняет действия от имени вызывающей стороны с меньшими полномочиями и без проверки.

В SPA для администрирования контента (2026-06-11) аудит нашёл три маршрута SW,
выполнявшихся с сохранённым админским токеном и **без проверки роли**:
`POST /api/github/org-role` (смена роли кому угодно), `POST /api/github/org-invite`
и отзыв приглашения. Маршруты roles-config *в той же директории* проверку имели.
То есть паттерн уже существовал, его просто не применили к более новым обработчикам. Один
`fetch('/api/github/org-role', {method: 'POST', body: '{"login":"me","role":"admin"}'})`
из любого контекста того же origin обходит всю модель RBAC.

## Как применять

Один общий шлюз, вызываемый первым в каждом привилегированном обработчике:

```ts
export const requireAdmin = (): Response | undefined => {
  const username = workerState.config?.username
  const role = username ? resolveRole(username) : undefined
  return role === 'admin' ? undefined : errorResponse('Admin only', 403)
}

// In each privileged handler:
return requireAdmin() ?? performPrivilegedThing(cfg, body)
```

Конструкция с `??` сохраняет обработчик декларативным: шлюз возвращает либо 403
`Response`, либо `undefined`, и реальная работа запускается только после того, как
шлюз пройден.

Важнее самого фрагмента два момента дизайна:

- **Проверка заново выводит права из состояния, которое вызывающая сторона задать не
  может.** Здесь роль выводится из файла ролей организации и из кеша админов
  организации. Она никогда не читает поле запроса или что-либо, что может нести
  postMessage.
- **GitHub под капотом всё равно проверяет scope токена.** Шлюз SW — это
  эшелонированная защита. Его задача — превратить «любой XSS означает захват
  организации» в «XSS ограничен тем, что текущий пользователь и так уже мог сделать».

## Антипаттерны

```ts
// "Only the admin UI calls this route."
// The SW cannot know that. Every same-origin script is a caller.
export const handleSetRole = async (request: Request) =>
  applyRole(cfg.owner, cfg.token, await request.json())

// Checking in the component instead of the handler:
v-if="role === 'admin'" // hides the button; the route still answers anyone
```

Симптом: ничего, пока кто-то враждебный это не найдёт. Проверки привилегий, живущие
только в шаблонах, не дают ни ошибок, ни логов, поэтому первым сигналом обычно
оказывается находка аудита.

## Контроль

Юнит-тест на каждый привилегированный маршрут, проверяющий 403 для вызывающей стороны
без прав администратора и успех для администратора, причём вызов GitHub замокан и
проверяется, что в ветке отказа он **не вызван**. Правило ревью на уровне grep:
каждый обработчик, читающий `config.token`, обязан либо вызывать шлюз, либо нести
комментарий с объяснением, почему он публичный.
