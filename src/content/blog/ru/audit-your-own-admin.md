В админке были OAuth с PKCE, файл ролей, параметризованные запросы к D1,
токены отписки с HMAC-подписью и сравнением за постоянное время, JWT, которые
проверяли подпись перед декодированием. По меркам чек-листа это аккуратная
кодовая база. И всё же структурная проверка за один вечер нашла три критичных
дыры, а закономерность за ними важнее самих находок: **каждая из них — это
граница, которая сдвинулась уже после того, как код вокруг неё был написан.**

## Прокси, потерявший свою модель угроз при переезде в порт

Браузеры не умеют разговаривать с GitHub по git smart-HTTP, потому что на тех
эндпоинтах нет CORS-заголовков, поэтому админка проксирует git-трафик через
крошечный воркер-маршрут. Изначально это был отдельный Cloudflare Worker с
зашитым пином хоста `github.com` и проверкой `ALLOWED_ORIGINS`. В какой-то
момент прокси переехал внутрь приложения — в Hono-маршрут на том же воркере,
который раздаёт SPA.

При переносе сохранили двадцать строк, которые делали работу, и выбросили
десять, которые отвечали за безопасность. Выложенный маршрут собирал
`https://${path}` прямо из пути запроса, отражал любой `Origin` и пересылал
каждый заголовок (включая `Authorization`) на тот хост, который назван в пути.
Это открытый релей с пересылкой учётных данных, висящий на боевом домене —
[ровно то, чем отдельная версия специально не должна была
быть](/principles/platform/proxy-must-pin-targets).

Никто не принимал решения убрать список разрешённых хостов. Он испарился при
переводе, потому что перенос рассматривали как «та же фича, новое место», а не
как новую поверхность атаки. Код, который существует ради соблюдения границы,
не переживает рефакторинг сам собой. Он выживает, когда граница записана там,
где рефакторингу придётся с ней столкнуться.

## Превью, доверявшее своим авторам

Превью редактора отдавало вывод `marked` прямо в `v-html`, и во всём дереве
зависимостей нигде не было санитайзера. Рефлекторное возражение здесь — «это же
наш собственный репозиторий с контентом, вход доверенный», — и оно рассыпается,
стоит посмотреть, кто пишет и кто читает. Редакторы (самая низкая роль) пишут
посты. Главные редакторы и админы просматривают их в этом самом превью, сидя в
сессиях, чей GitHub-токен несёт `admin:org`.

То есть граница привилегий проходит ровнёхонько через середину фичи, и
[вывод markdown — это HTML
злоумышленника](/principles/platform/sanitize-html-before-injection) по неправильную
сторону этой границы. Один `<img onerror>` в черновике — и сам процесс ревью
доставляет org-admin токен тому, кто попросил проверить. Чинится это одним
вызовом DOMPurify в точке вставки плюс тестовым файлом, перечисляющим векторы.
Стоит почти ничего. Сложным было только заметить, что «доверенный контент»
тихо превратился в «контент с более низкого уровня привилегий».

## Service Worker, делавший всё, о чём попросят

SW хранит токен пользователя и открывает привилегированные маршруты: сменить
роль пользователя в организации, отправить приглашение. Соседние маршруты
конфигурации ролей проверяли роль вызывающего. Эти — нет. Любой, у кого есть
выполнение скрипта в том же origin, мог отправить POST с самоповышением до
админа, и SW — [классический запутавшийся
заместитель](/principles/platform/confused-deputy-in-the-service-worker) — подписал бы
этот запрос сохранённым админским токеном.

В связке с XSS в превью это превращается в «из редактора в админа организации»
одним подготовленным постом. Любая находка по отдельности — это локальная
ошибка, но именно цепочка сделала вечер не зря потраченным. Проверки, которые
останавливаются на «нашли XSS, завели тикет», упускают, что серьёзность находки
N зависит от находок с N+1 по N+3.

## Скучный слой тоже был сломан

В CI было ноль блоков `permissions:` на девять воркфлоу в двух репозиториях,
каждый action был запинён на изменяемый тег, а в env уровня джобы лежал PAT на
триггере `pull_request`. Ничего экзотического. Это [состояние GitHub Actions по
умолчанию](/principles/build-ci-deploy/least-privilege-workflows) — именно поэтому оно
встречается повсюду. Чинится механически: один блок permissions, пины по SHA,
dependabot, чтобы они двигались, секреты с областью видимости на уровне шага за
проверкой «тот же репозиторий». Заняло меньше времени, чем описать это.

Ещё один умолчательный момент стоит назвать. Ссылка отписки переключала
состояние подписчика по GET. Почтовые клиенты предзагружают ссылки, а
антивирусные сканеры по ним ходят, так что от тихой отписки людей отделял один
проход Outlook safe-links. GET-обработчик, который что-то меняет, — это класс
багов, а не вопрос стиля.

## Что реально обобщается

- **Перепроверяйте после переезда.** Прокси переехал внутрь приложения,
  обработчик скопировали под новый маршрут, поток авторизации перенесли в
  попап — каждый перенос заново открывает вопросы, на которые оригинал уже
  ответил. Сравнивайте *защиты*, а не только поведение.
- **Записывайте модель угроз там, где код.** Пин хоста, который умер при
  переносе, выжил бы как модуль `cors-allow.ts` с тестами, проверяющими ответ
  403. Тесты — единственная документация, которую рефакторинг обязан прочитать.
- **Метки доверия гниют.** «Доверенный контент», «внутренний эндпоинт», «наш
  собственный репозиторий» — каждая такая фраза в комментарии это TODO:
  проверь, кто на самом деле сегодня стоит по обе стороны границы.
- **Проверяйте параллельно, чините последовательно.** Три прицельных прохода
  (поверхность сервера, клиент/авторизация, CI/цепочка поставок) нашли за часы
  то, что один универсальный проход размазал бы на дни. Правки потом легли одним
  отревьюенным PR с тестом на каждую находку.

Неудобный вывод самопроверки — никогда не «код был неряшливым». А в том, что код
верно сохранил решения, чей контекст уже истёк. Гигиена по чек-листу держалась
везде: параметризованные запросы, сравнения за постоянное время, подписанные
токены. Сломалось всё то, что *сдвинулось*.
